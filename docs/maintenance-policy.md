# Maintenance Policy

This is the maintainer-facing policy. Contributor-facing rules live in [CONTRIBUTING.md](../CONTRIBUTING.md).

## Source Of Truth

Structured YAML in `data/` is the source of truth. `README.md` and `docs/COMPARISON.md` are generated by `npm run generate` and must never be edited by hand.

## Sorting

Tools are sorted by display name using English locale comparison, then by slug as a tiebreaker. Tool category, tag, interface, and source arrays are normalized alphabetically. `primary_category` is preserved as-is during sorting; it is not sorted into the category array.

```bash
npm run sort
```

## Primary Category

Reviewed tools that belong to more than one category must set `primary_category` to one of the slugs in their `categories` list. This controls which shelf is shown in the comparison matrix and the review queue. Single-category tools do not need it.

## Validation

`npm run validate` (also part of `npm test`) is fully offline and deterministic; it never depends on external network access. It blocks:

- malformed or missing required fields, invalid URLs, invalid dates, invalid enum values
- duplicate slugs and names
- unrecognized category/tag references on reviewed entries
- descriptions outside 40-180 characters
- promotional description language (superlative or hype phrases, exclamation marks, emoji)
- citation or research artifacts in public fields
- invalid SPDX license expressions
- unsorted `data/tools.yml`, stale `README.md`, and stale `docs/COMPARISON.md`

It warns (without failing) about:

- soft subjective description wording (for example "intuitive", "robust", "enterprise-grade")
- reviewed entries whose sources are not official or known evidence hosts
- entries sharing the same normalized `website_url` or `repo_url` (usually a draft facet of an already reviewed tool; merging is a curation decision)

Freshness comparisons ignore CRLF/LF differences, so Windows checkouts validate correctly.

```bash
npm test
```

## Scheduled Maintenance

Network-dependent checks run in the weekly `Catalog maintenance` workflow (`.github/workflows/check-links.yml`), also triggerable manually. Pull-request validation stays offline.

### Link checking

```bash
npm run check:links                     # full catalog
npm run check:links -- --only github.com/example   # scoped run
```

Each URL gets one of three verdicts:

- `ok` — some request returned HTTP 2xx (HEAD falls back to GET automatically).
- `broken` — HTTP 404/410 or unresolvable DNS, confirmed on a second attempt. Only this fails the run (exit 1). Fix or remove these URLs.
- `inconclusive` — rate limits, server errors, timeouts, bot walls, TLS problems after bounded retries. Reported for follow-up; never fails the run.

### Stale entries

```bash
npm run check:stale                          # default threshold: 180 days
npm run check:stale -- --max-age-days 90     # custom threshold
```

Reports entries whose `last_checked` is older than the threshold, plus the ten oldest entries. Reporting always exits 0 — entries aging naturally must not fail CI. `--fail-on-stale` opts into a hard failure for stricter runs. After re-verifying an entry against its sources, set `last_checked` to the verification date.

## Review Policy

Reviewers should check that sources are official, descriptions are factual, uncertain fields are marked conservatively, and entries improve the directory rather than inflating counts.
